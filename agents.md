# Mellitrack — Agent Reference

## Overview
Mellitrack is a fitness tracking web app. Users log training sessions (date + exercises with sets of weight/reps). Data is visualized per-exercise and per-category on dashboards.

## Tech Stack
- **Frontend**: React 18 + Vite + Tailwind CSS + React Router + Recharts
- **Backend**: Hono framework running on Cloudflare Workers
- **Database**: Cloudflare D1 (SQLite)
- **Auth**: Session cookie (httpOnly) with SHA-256 password hashing
- **Deployment**: Cloudflare Workers with static assets inlined into the worker bundle

## Project Structure
```
src/
  client/           React SPA
    api.ts          API client (fetch wrappers)
    pages/          Route components (Dashboard, TrainingForm, etc.)
  server/           Hono worker
    index.ts        Entry point, auth middleware, static serving
    routes/         Hono sub-routers (auth, exercises, trainings, progress)
  server/static-manifest.ts   Auto-generated by build — inlined static assets
scripts/
  inline-static.ts  Build script that inlines dist/ into static-manifest.ts
  seed.sql          Seed data for local dev
e2e/
  basic.spec.ts     Playwright E2E tests
  global-setup.ts   Seeds local D1 before test run
migrations/
  0001_initial.sql  Schema (users, exercise_categories, exercises, trainings, training_exercises, sets, exercise_category_mappings)
```

## Key Commands
```bash
# Dev (local D1 + Vite dev server)
npm run dev

# Type check only
npm run typecheck

# Build (generates static-manifest.ts from dist/)
npm run build

# Deploy to production
npm run deploy

# Local DB
npm run db:migrate       # apply migrations to local D1
npm run db:migrate:prod  # apply migrations to prod D1
npm run db:seed          # seed local D1 with test data

# Tests
npm run test:e2e         # headless Playwright
npm run test:e2e:ui      # Playwright UI mode
```

## Architecture Notes
- **Static assets**: Not served from KV. The build script (`scripts/inline-static.ts`) reads `dist/` and generates `src/server/static-manifest.ts`, which the worker imports and serves from memory. This avoids KV manifest issues.
- **D1**: Single SQLite database. Migrations are SQL files in `migrations/`. Always create a migration for schema changes — never ALTER TABLE directly on prod. Apply with `npm run db:migrate` (local) or `npm run db:migrate:prod` (prod).
- **Auth**: `session` cookie contains user ID. Middleware at `src/server/index.ts` verifies it against the `users` table.
- **Prefill logic**: When creating a new training for a category, the form prefills each exercise's sets from the *last training of that same category* for that exercise. This is handled by `GET /api/trainings/last-category/:categoryId`.
- **Tombstoning**: Exercises have a `deleted_at` column. Unreferenced exercises are hard-deleted; referenced ones are soft-deleted (tombstoned). Tombstoned exercises don't appear in new training forms but remain in historical data.
- **Reorder**: Categories and exercises have a `sort_order` column. Up/down arrows in the UI call `PUT /api/exercises/categories/reorder` and `PUT /api/exercises/reorder` with the new ID order.

## GitHub Actions Workflows

All workflows use `CLOUDFLARE_API_TOKEN` and `CLOUDFLARE_ACCOUNT_ID` secrets.

| Workflow | Trigger | What it does |
|---|---|---|
| `ci.yml` | PR to `main`, push to `main` | Runs `typecheck`, `build`, then `e2e` (Playwright with local D1) |
| `preview-deploy.yml` | PR opened/synchronized | Forks prod D1 via backup/restore, applies migrations, deploys a preview worker (`mellitrack-pr-N`), comments URL on PR |
| `preview-cleanup.yml` | PR closed | Deletes preview worker and D1 database, strikes out preview URL comment |
| `prod-deploy.yml` | Push to `main` | Runs migrations on prod D1, builds, deploys worker |

## Environment Variables / Secrets
- `CLOUDFLARE_API_TOKEN` — Wrangler/Cloudflare API token (needs Workers, D1, KV permissions)
- `CLOUDFLARE_ACCOUNT_ID` — Cloudflare account ID
- These are used by GitHub Actions and local `wrangler` commands.
